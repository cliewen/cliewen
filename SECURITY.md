# Security Policy

## Supported Versions

Security fixes are made for the current development line and the latest published release.

| Version | Supported |
|---|---|
| `main` | Yes |
| Latest release | Yes |
| Older releases | No; upgrade to the latest release |

## Reporting a Vulnerability

Report suspected vulnerabilities privately by emailing [flemming.n.larsen+cliewen-security@gmail.com](mailto:flemming.n.larsen+cliewen-security@gmail.com) with the subject `[Cliewen Security] <short vulnerability summary>`.

Do not open a public issue, pull request, or discussion for a suspected vulnerability.

Please include:

- The affected Cliewen version or commit and how it was installed
- The expected security boundary and potential impact
- Reproduction steps or a proof of concept
- Any known mitigations or suggested fix

Avoid including credentials, personal information, or unrelated sensitive data. Reports about the `clue` CLI, shipped skills, release artifacts, and repository workflows are in scope; ordinary defects belong in the public bug form.

## What to Expect

We aim to acknowledge a report within 7 calendar days and provide an initial status within 14 calendar days. The initial status may include a request for more information, an assessment of severity and scope, or a proposed next step.

Remediation time depends on severity, complexity, and release risk, so we do not promise a fixed resolution date. We will share material status changes with the reporter and coordinate disclosure timing when a vulnerability is confirmed.

Please keep the report confidential until we agree that disclosure is safe. We will credit reporters who want public recognition and will respect requests to remain anonymous.

GitHub private vulnerability reporting may become an additional channel after the repository is public. Email remains the supported reporting route.
